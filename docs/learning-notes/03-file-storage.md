# 학습 노트 03: 파일 저장소

## 핵심 요약

작업 7의 목표는 메모리에만 존재하던 `Row`를 디스크 파일에 저장해서 **영속성**을 얻는 것이다.

이전 작업에서는 행을 바이트 배열로 바꾸는 직렬화를 구현했다.

```text
Row -> byte[]
byte[] -> Row
```

이번 작업에서는 그 바이트 배열을 파일에 기록하고, 다시 파일에서 읽어 행으로 복원한다.

```text
Row -> byte[] -> file
file -> byte[] -> Row
```

즉, 작업 7은 `RowSerializer`로 만든 바이트 표현을 실제 디스크 저장소에 연결하는 단계다.

---

## 저장 모델

현재 저장소는 매우 단순한 구조를 가진다.

```text
[row 0][row 1][row 2][row 3]
```

파일은 여러 개의 행을 연속해서 저장한다. 각 행은 `Schema`를 기준으로 계산된 고정 크기 바이트 배열이다.

예를 들어 현재 스키마는 다음과 같다.

```text
id   INT          4바이트
name VARCHAR(32) 32바이트
```

따라서 행 하나의 크기는 36바이트다.

```text
rowSize = 4 + 32 = 36
```

이 구조에서 파일은 “완전한 행들의 연속”으로 해석된다.

---

## 고정 길이 레코드와 오프셋 계산

고정 길이 레코드의 핵심 장점은 **파일 오프셋 계산이 단순하다**는 점이다.

행 하나의 크기가 항상 `rowSize`라면 n번째 행의 시작 위치는 다음처럼 계산할 수 있다.

```text
rowOffset = n * rowSize
```

예를 들어 `rowSize = 36`이면 다음과 같다.

```text
row 0 시작 위치 = 0
row 1 시작 위치 = 36
row 2 시작 위치 = 72
row 3 시작 위치 = 108
```

가변 길이 레코드라면 n번째 행을 찾기 위해 앞 행들의 길이를 알아야 한다. 반면 고정 길이 레코드는 `rowSize`만 알면 원하는 행의 위치를 바로 계산할 수 있다.

대신 공간 낭비가 생길 수 있다. 예를 들어 `VARCHAR(32)` 영역에 `"kim"`을 저장하면 실제 값은 3바이트지만, 저장 공간은 32바이트를 차지한다. 남는 공간은 0으로 채운다.

```text
"k" "i" "m" 0 0 0 ... 0
```

정리하면 다음과 같다.

```text
장점: 위치 계산이 쉽다.
단점: 짧은 값도 고정 크기 영역을 차지하므로 공간이 낭비될 수 있다.
```

## 순차 쓰기와 append-only 구조

이번 작업의 쓰기는 파일 끝에 새 행을 붙이는 방식이다.

```text
[row 0][row 1][row 2][new row]
```

기존 행의 위치는 바꾸지 않는다. 파일 중간에 데이터를 끼워 넣지도 않고, 빈 공간을 찾지도 않는다.

이 방식은 append-only 구조에 가깝다.

장점은 단순성이다.

```text
새 행은 항상 파일 끝에 쓴다.
기존 행의 위치를 변경하지 않는다.
빈 공간 관리가 필요 없다.
중간 삽입이나 재배치를 고려하지 않아도 된다.
```

하지만 한계도 명확하다.

```text
삭제한 공간을 재사용하지 못한다.
수정 후 남거나 부족한 공간을 처리하기 어렵다.
파일이 계속 커질 수 있다.
특정 구간만 관리할 저장 단위가 없다.
```

따라서 append-only 저장은 영속성을 배우기 위한 첫 단계로는 적합하지만, 삭제, 수정, 빈 공간 재사용까지 다루기에는 부족하다.

## 파일 불변 조건

이번 작업에서 중요한 불변 조건은 다음이다.

```text
파일 크기는 rowSize의 배수여야 한다.
```

파일은 완전한 행들의 연속이어야 한다. `rowSize = 36`이라면 정상적인 파일 크기는 다음과 같다.

```text
36바이트  = 행 1개
72바이트  = 행 2개
108바이트 = 행 3개
```

반면 파일 크기가 73바이트라면 다음과 같이 해석된다.

```text
72바이트 = 완전한 행 2개
1바이트  = 해석할 수 없는 남은 데이터
```

마지막 1바이트는 완전한 행이 아니므로, 파일이 깨졌거나 쓰다 만 상태라고 볼 수 있다.

따라서 `readAll`은 파일을 읽기 전에 파일 크기가 `rowSize`의 배수인지 검사한다. 이 검사는 파일을 안전하게 행들의 연속으로 해석하기 위한 최소한의 방어선이다.

## 메모리 상태와 디스크 상태

현재 구조에는 두 가지 상태가 있다.

```text
Table   = 실행 중 빠른 조회를 위한 메모리 상태
RowFile = 재시작 후에도 남는 디스크 저장소
```

`Table`은 메모리에 있으므로 조회가 빠르다. `select`를 실행할 때마다 파일을 다시 읽지 않고, 현재 실행 중인 `Table`에서 바로 행을 가져온다.

`RowFile`은 디스크 파일에 데이터를 저장한다. 디스크는 메모리보다 느리지만, 프로그램이 종료되어도 데이터가 남는다.

그래서 `insert`가 실행되면 현재 구조에서는 두 곳을 모두 갱신한다.

```text
Table에 추가   -> 현재 실행 중 select 결과에 바로 반영된다.
RowFile에 추가 -> 프로그램을 껐다 켜도 데이터가 남는다.
```

이 구조는 아직 버퍼 풀은 아니지만, 메모리 상태와 디스크 상태의 역할 차이를 보여준다.

## 이번 작업에서 구현한 것

이번 작업에서 추가한 핵심 클래스는 `RowFile`이다.

`RowFile`의 책임은 다음과 같다.

```text
append(row)
- Row를 RowSerializer로 byte[]로 바꾼다.
- 파일 끝에 byte[]를 추가한다.

readAll()
- 파일 전체를 읽는다.
- rowSize 단위로 byte[]를 끊는다.
- 각 byte[]를 Row로 복원한다.
```

추가로 다음 조건을 처리한다.

```text
파일이 없으면 빈 목록을 반환한다.
파일 크기가 rowSize의 배수가 아니면 오류로 처리한다.
```

## 현재 구조의 한계

현재 파일 저장소는 행을 단순히 연속으로 저장한다.

```text
[row][row][row][row]
```

이 구조는 이해하기 쉽지만, 저장소로서는 아직 한계가 있다.

첫째, 저장 공간의 관리 단위가 없다. 파일 전체는 행들의 연속일 뿐이고, 특정 구간을 페이지처럼 독립적으로 관리하지 않는다.

둘째, 삭제와 수정이 어렵다. 삭제된 행의 공간을 어떻게 표시할지, 그 공간을 나중에 어떻게 재사용할지에 대한 구조가 없다.

셋째, 메모리에 올릴 단위가 없다. 나중에 버퍼 풀을 만들려면 디스크에서 읽어 메모리에 캐시할 고정 단위가 필요하다.

정리하면 현재 구조의 한계는 다음과 같다.

```text
빈 공간 관리가 없다.
삭제 표시가 없다.
수정 시 공간 재사용 전략이 없다.
디스크 읽기/쓰기 단위가 없다.
버퍼 풀과 연결할 캐시 단위가 없다.
```

## 다음 작업과의 연결

작업 7은 영속성을 얻는 첫 단계다. 하지만 행을 파일에 단순히 이어 붙이는 구조만으로는 데이터베이스 저장소를 확장하기 어렵다.

다음 단계에서는 파일을 페이지 단위로 나눈다.

```text
[page 0][page 1][page 2]
```

페이지는 여러 행을 담는 고정 크기 블록이다. 페이지가 생기면 디스크 읽기/쓰기 단위가 생기고, 이후 버퍼 풀에서 메모리에 올릴 단위도 생긴다.

또한 페이지 안에서 행 위치, 빈 공간, 삭제 표시 등을 관리할 수 있다.

작업 7의 결론은 다음과 같다.

```text
Row를 고정 길이 바이트로 직렬화해서 파일 끝에 순차적으로 저장하면 영속성을 얻을 수 있다.
하지만 행을 파일에 단순히 이어 붙이는 구조만으로는 삭제, 수정, 빈 공간 관리가 어렵다.
그래서 다음 단계에서는 페이지라는 디스크 관리 단위가 필요하다.
```
